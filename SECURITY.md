# Security Policy

## Reporting a vulnerability

If you find a security issue in Xova, please report it **privately** — not in a public GitHub issue.

**Email:** adzextreem@gmail.com
**Subject line:** `[Xova security]` followed by a one-line summary.

Please include:

- A description of the issue
- Steps to reproduce
- The version (`xova --version` or check `Settings → ⚙` → `/version` in the chat)
- Your assessment of impact and severity, if you have one

I'll acknowledge within 7 days and follow up with a fix or a clear "won't fix" reason within 30 days.

If the issue is already public knowledge (e.g. it's already on Hacker News), a public issue is fine — at that point disclosure has already happened.

## What counts

Yes:
- Local privilege escalation triggered by Xova (e.g. opening admin terminals when not asked)
- Code execution via crafted chat input
- Path-traversal in `xova_read_file` / `xova_write_file` / `xova_run`
- Credential / token leakage in logs, backups, or chat exports
- Tauri WebView sandbox escapes specific to Xova's commands
- Voice command injection (e.g. crafted audio that bypasses speaker recognition)

Probably not:
- Bugs that require an attacker to already have full local access (that's already game over)
- Issues in Ollama, Tauri, Whisper, Resemblyzer, or any other dependency — please report those upstream and let me know so I can pin a safe version
- Reports generated by a vulnerability scanner without manual verification

## Scope

This policy covers:

- The Xova desktop app (`xova.exe`, the Tauri/Rust backend, the React frontend)
- The accompanying `backup_xova.ps1` script
- The cross-AI bridge files in `C:\Xova\memory\` (specifically: how Xova reads/writes them, not what other apps put there)

Out of scope:

- The Jarvis daemon (separate repo at `C:\jarvis\` — report there)
- The `recursive-field-math-pro` library and other Recursive Field Framework repos (report on those individually)
