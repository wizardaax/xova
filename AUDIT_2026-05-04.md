## CRITICAL (5)

### CRIT-1 — Unrestricted filesystem access exposed as LLM tools
**Files:** `lib.rs` lines 577 (`xova_read_file`), 582 (`xova_list_dir`), 594 (`xova_write_file`), 604 (`xova_delete_path`)

All four Tauri commands perform raw `std::fs` operations on any path the caller passes. The LLM tool schema registered at line 440-458 includes `xova_read_file` with the description "No path restrictions." The 3B model can read SSH keys, `.env`, `jarvis.db`, Windows credential vaults, and anything accessible by `xova.exe`'s process token. `xova_write_file` creates parent directories and overwrites arbitrary paths. `xova_delete_path` calls `remove_dir_all` or `remove_file` unconditionally. `xova_list_dir` traverses any directory. A prompt injection in any document Adam uploads is sufficient to trigger these. The description "No path restrictions" is literally in the schema that gets sent to the model on every non-`disableTools` call.

### CRIT-2 — Arbitrary shell execution + UAC elevation via LLM tool
**File:** `lib.rs` lines 1136-1181 (`xova_run`)

`fn xova_run(command: String, ...)` passes the command string directly to `cmd /C <command>`. No filtering, no allowlist, no escaping. `elevated=true` launches `cmd.exe` via `Start-Process -Verb RunAs`, which triggers UAC elevation. The LLM tool schema says "Run any shell command via cmd.exe." A single prompt-injection sentence in a pasted document or a model hallucination about what command to run can result in `xova_run` executing arbitrary shell code with an admin prompt.

---

### CRIT-3 — EvolutionEngine silently rewrites Python source files with no backup and no approval
**File:** `mesh_runner.py` lines 174-279 (`_try_apply_patch`), line 344 (call site)

When `human_gate=False` on an EvolutionEngine change proposal, `_try_apply_patch()` is called immediately. It opens Python files under `C:\jarvis\src`, `C:\Xova`, and `D:\github\wizardaax` using `os.walk`, matches on filename stem, reads lines, applies regex key=value replacements, and writes them back with `open(fp, "w")`. No deposit-to-trash, no backup, no user approval, no git commit. The patch spec JSON is written (good for provenance) but the actual source modification is silent. Currently dormant because `mesh_flags.json` has `evolutionEnabled: false`, but the mechanism fires the instant that flag is toggled true. This directly violates RULE 2 (never delete/modify without asking) and RULE 4 (deposit-to-trash before any data-loss operation).

### CRIT-4 — Confirmed mesh_runner crash loop — no backoff, no circuit breaker
**Files:** `watchdog.log` lines 1-44, `xova_watchdog.py` lines 155-157, `mesh_runner.py` lines 15-18

watchdog.log shows mesh_runner spawned 21+ times between 03:21 and 03:23 — one new process every ~6 seconds — each one dying before the next poll cycle. The root cause: `mesh_runner.py` line 15 does `sys.path.insert(0, r"D:\github\wizardaax\Snell-Vern-Hybrid-Drive-Matrix\src")` and line 18 does `from snell_vern_matrix.agents.cognitive_cycle import CognitiveCycle`. If that import fails (path missing, package not installed, broken dep), the process exits immediately. The watchdog has no crash counter, no exponential backoff, no "stop retrying after N failures" logic. Result: 21 dead `pythonw.exe` processes in 2 minutes, CPU churn from repeated Python interpreter startup, orphaned processes accumulating.

### CRIT-5 — `xova_ask_claude` passes full memory context as a single CLI argument — hits Windows 32KB limit
**File:** `lib.rs` lines 1228-1250

`fn xova_ask_claude` builds `full_prompt` by prepending all `project_*.md` and `feedback_*.md` files from the Forge memory directory (described in comments as ~30KB) to the user's question, then passes it as `cmd.arg(&full_prompt)`. Windows `CreateProcess` has a 32,767-character command-line limit. When the concatenated prompt exceeds that limit, `CreateProcess` returns `ERROR_FILENAME_EXCED_RANGE` or silently truncates. The `/ask` command either fails with a cryptic error or silently drops context without any warning to Adam.

### HIGH-1 — `xova_ask_jarvis` Tauri command writes incomplete schema — `intent`, `to`, `correlation_id` missing
**File:** `lib.rs` lines 1114-1133; `xova_inbox.py` lines 82-91

The Tauri command writes `{role, text, ts, from:"xova"}` only. The XovaInboxListener reads `to_field`, `correlation_id`, and `intent` from the inbox file. Without `to_field`, the `to_field == "xova"` routing check at `xova_inbox.py` line 91 never fires for Tauri-originating messages. Without `correlation_id`, reply matching in `ask_xova.py` falls back to ts-based matching exclusively. The richer schema visible in the current `jarvis_inbox.json` on disk comes from a different write path (not the Tauri command). Every LLM tool call of `xova_ask_jarvis` from App.tsx line 4087 uses the minimal schema.

### HIGH-2 — Single-file inbox bridge has no queue — concurrent writes silently discard earlier messages
**Files:** `ask_xova.py` lines 67-79; App.tsx bridge poller lines 822-921

`xova_chat_inbox.json` is a single-writer-wins file. If Jarvis's `AskXovaTool` writes `{ts:T1, text:"Q1"}` and then Forge writes `{ts:T2, text:"Q2"}` before Xova's 2-second poll tick processes the first, Q1 is silently discarded. `ask_xova.py`'s poll at line 84-103 checks `in_reply_to_ts == ts` or `correlation_id` match — it will wait 30 seconds for a reply that will never come, then time out. The same race applies to all six bridge files (`jarvis_inbox.json`, `voice_inbox.json`, `xova_chat_inbox.json`, `xova_chat_outbox.json`, `xova_command_inbox.json`, `xova_slash_inbox.json`). None use atomic writes, file locking, or append-log semantics.

### HIGH-3 — `/facts` and `/standing-facts`/`/known` read incompatible formats from the same file path — mutual destruction
**File:** `App.tsx` lines 1730-1741 vs lines 2653-2675

`/facts` (line 1730) reads React state `standingFacts` (a `string[]`) that was persisted via `saveMemory("xova_standing_facts", string[])` — the file on disk is double-JSON-encoded `"[\"fact1\",...]"`. `/standing-facts` and `/known` (line 2654) call `xova_read_file("xova_standing_facts.json")` expecting `{user_facts:[], world_facts:[], directives:[], synced_at:...}` written by `/sync-facts`. Running `/consolidate` after `/sync-facts` overwrites the structured object with a string array — `/standing-facts` then gets an array where it expects an object, `.user_facts` is `undefined`, it shows empty. Running `/sync-facts` after `/consolidate` overwrites the double-encoded array with the object — `loadMemory<string[]>("xova_standing_facts")` parses a JSON object, `Array.isArray` returns false, `standingFacts` stays empty, `/facts` shows "No standing facts yet." The second command always silently destroys the first command's output.

### HIGH-4 — `xova_status` is a synchronous blocking Tauri command called every 5 seconds
**File:** `lib.rs` lines 940-999; App.tsx line 362

`fn xova_status()` is not declared `async`. It calls `reqwest::blocking::Client` with a 3-second timeout to `localhost:11434/api/ps`, then spawns PowerShell to query process state, then runs `nvidia-smi`. Potential blocking time per call: 3s (Ollama timeout) + 1-2s (PowerShell) + 0.5s (nvidia-smi) = ~4-6s. Tauri's `#[tauri::command]` without `async` occupies a Tokio worker thread for the duration. The 5-second setInterval at App.tsx line 362 means status calls are in-flight during every LLM inference cycle. When the async Ollama streaming command (`ollama_chat_stream`) is running and a status poll fires simultaneously, both compete for the same Tokio runtime thread pool. On a 4-thread pool this degrades streaming responsiveness.

### HIGH-5 — Watchdog unconditionally kills all Jarvis processes before restart — destroys in-memory dialogue state
**File:** `xova_watchdog.py` lines 97-120

`_start_jarvis()` calls `_find("jarvis.daemon")` to get existing PIDs, kills them all with `taskkill /F`, waits 1 second, then starts a fresh daemon. This is called whenever `_find("jarvis.daemon")` returns empty during the poll loop — i.e., whenever Jarvis is not detected. The detection uses `Get-CimInstance Win32_Process` via PowerShell; if that query is slow or the `pythonw.exe` CommandLine field is temporarily unavailable, `_find()` returns `[]` for a live process. The watchdog then calls `_start_jarvis()` on a running Jarvis, kills it, and starts a new one. Watchdog.log confirms this pattern: Jarvis killed and restarted at 03:26, 03:34, 04:03, 04:39, 04:45, 04:46 — all while xova.exe was alive. Per RULE 3, each kill wipes in-memory dialogue context, hot Whisper model state, and in-flight voice requests.

### HIGH-6 — `start_xova.ps1` independently launches mesh_runner AND launches watchdog — double-start race creating duplicate writers
**File:** `start_xova.ps1` lines 110-120; `xova_watchdog.py` lines 155-157

The PowerShell launcher starts mesh_runner directly via `Start-Process pythonw.exe C:\Xova\mesh_runner.py` and also starts xova_watchdog.py. The watchdog then independently starts mesh_runner again within the first 5-second poll if it doesn't find an existing one. If the ps1-launched runner starts and crashes (the crash loop), the watchdog relaunches it — correctly. But if the ps1 runner starts and survives, the watchdog may not yet have found it and starts a second one. Two instances write to `mesh_feed.jsonl` concurrently with no file lock. The `_append()` function (mesh_runner.py line 94-97) does a bare `open(FEED_PATH, "a")` with no locking — concurrent writes produce interleaved or corrupted JSONL lines.

### HIGH-7 — `ollamaChatStream` result inconsistency — placeholder not cleared on tool_calls path
**File:** `App.tsx` lines 3836-3848, 4118-4126

During streaming the `streamed` variable accumulates tokens and updates the placeholder bubble live. If the final result type is `"tool_calls"` (line 3936), the code dispatches tools, calls `ollamaChat` for follow-up (line 4108), and then at line 4118-4120 replaces the placeholder with `finalText`. The issue: between the `ollamaChatStream` returning and the `ollamaChat` follow-up completing, the placeholder shows the raw partial streaming tokens (whatever the model emitted before the tool_call JSON). On the 3B model this is often empty, but when the model streams a preamble before its tool call (e.g., "Let me check that for you..."), that preamble stays visible in the placeholder for the entire tool execution duration — then gets overwritten by `finalText`. Produces a confusing mid-stream state for the user.

## MEDIUM (10)

### MED-1 — `save_memory` double-JSON encoding — all Python readers must double-parse or silently get wrong type
**Files:** `mesh.ts` line 69; `lib.rs` line 286; `mesh_runner.py` line 83

`saveMemory(key, value)` calls `JSON.stringify(value)` before passing to Tauri. Tauri's `save_memory` writes the string as-is. So `C:\Xova\memory\mesh_flags.json` contains `"{\"evolutionEnabled\":false,...}"` — a JSON string wrapping JSON. `mesh_runner.py` handles this with a double-parse at line 83. `loadMemory` in TS does one `JSON.parse` which correctly recovers the object. Any new Python reader that does a naive `json.load(f)` gets a string, not the expected object, and produces silent wrong behavior. The architecture is functional but fragile for new consumers.

---

### MED-2 — `/banter` has a hardcoded 12-second unconditional wait that isn't interruptible
**File:** `App.tsx` line 1133

`await new Promise((r) => setTimeout(r, 12000))` is an unconditional 12-second block in a slash handler that does NOT set `isBusy`. The Stop button is not active during banter. `cancelledRef.current` is not checked after the wait. If Jarvis's fast-path reply arrives in 2 seconds, Xova still waits 10 more seconds before delivering R3. If Jarvis never replies (Jarvis down), Xova still fires R3 after 12 seconds regardless. The 12-second wait is a proxy for "Jarvis replied" that does not actually verify a reply arrived.

---

### MED-3 — `xova_vision` tool dispatch auto-takes a new screenshot unconditionally before vision call
**File:** `App.tsx` lines 4034-4037

When the LLM calls `fn === "xova_vision"`, the first action is `await invoke("xova_computer", { action: '{"cmd":"screenshot"}' })`, taking a fresh screenshot of the current screen state. Then `xova_vision` is called with `args.image_path || "C:\Xova\memory\screen.png"`. If the model called `xova_vision` with a specific `image_path` pointing to an uploaded file, the auto-screenshot is taken unnecessarily (wasted call). More problematically, after the `setTimeout(r, 500)` delay (line 4036), the screen state has changed. The practical impact is low since `args.image_path` is used directly for vision — but the wasted screenshot pollutes `screen.png` with a different frame than what was visible when the model decided to invoke vision.

### MED-4 — `mesh_feed.jsonl` grows without any size cap
**File:** `mesh_runner.py` `_append()` line 94-97

`_append()` opens in `"a"` mode and writes lines indefinitely. No rotation, no size limit, no line cap. The file already contains 12,578+ entries. At ~10-13 lines per 60-second cycle when enabled, that's ~850 lines per hour, ~20,000 per day. The MeshFeed component in App.tsx reads this file on every feed update. `forge_events.jsonl` is capped at 500 lines (App.tsx line 117-118) but `mesh_feed.jsonl` is not. When the mesh runner is re-enabled, this file will grow until it causes slow reads or file system issues.

### MED-5 — `logForgeEvent` read-modify-write pattern for `forge_events.jsonl` is not atomic
**File:** `App.tsx` lines 108-119

`logForgeEvent` reads the full file via `xova_read_file`, appends the new entry in memory, trims to 500 lines, and writes the full file back via `xova_write_file`. If two async calls to `logForgeEvent` interleave — which is very likely since self-eval fires async at line 3856 and auto-correction fires async at line 3904, both potentially concurrent — the second read sees stale content, the second write overwrites the first write's entry. Lost log entries, not data corruption in the dangerous sense, but the 500-line cap and event ordering are unreliable under concurrent logging.

### MED-6 — Auto-consolidation fires on `messages.length % 40 === 0` — triggers every message after 80, 120, 160...
**File:** `App.tsx` lines 3808-3809

`if (messages.length >= 40 && messages.length % 40 === 0)` fires at 40, 80, 120, 160... which is the intended "once per 40 messages" behavior. But `messages` is captured from the `onSend` closure's render — if `onSend` runs multiple times quickly (e.g., slash bridge dispatches in rapid succession), the modular condition may fire on every call while `messages.length % 40 === 0` holds, since React state updates are batched. In practice this is rare but the intent of "fire once per 40 messages" is not guaranteed by this arithmetic.

### MED-7 — `AskXovaTool.inputSchema` doesn't declare `correlation_id` — LLM-driven calls always use ts-matching
**File:** `ask_xova.py` lines 44-54, line 64

`correlation_id` is accepted in `run()` at line 64 but not declared in `inputSchema`. The LLM tool router will never populate it. All LLM-driven `askXova` calls use ts-based `in_reply_to_ts` matching, which fails under concurrent requests. Only code-driven calls from `xova_inbox.py _call_ask_xova_direct()` pass a `correlation_id`. The schema should declare the field to allow the LLM to use it when forwarding structured requests.

### MED-8 — `xova_ask_jarvis` Tauri command is synchronous despite performing file I/O
**File:** `lib.rs` line 1114

`fn xova_ask_jarvis(...)` is not declared `async fn`. It performs `fs::write(...)` synchronously on a Tauri thread. File writes on Windows that go through SmartScreen or antivirus scanning can block for hundreds of milliseconds. Per Tauri 2 guidelines, commands that do I/O should be async. Low impact given how fast this write is in practice, but inconsistent with all other I/O commands.

`const [chatMode, setChatMode] = useState<"xova" | "team">("xova")` — no branch in `onSend`, the LLM system prompt, or any UI element distinguishes `chatMode === "team"` from `"xova"`. Dead state. If a UI element sets `chatMode` to `"team"`, nothing changes in behavior.

### MED-10 — `recallTokens` regex minimum 3 chars makes some `RECALL_STOPWORDS` entries redundant; duplicates present
**File:** `App.tsx` lines 84-94

`RECALL_STOPWORDS` contains entries like `"are"`, `"for"`, `"not"`, etc. (3-char words that ARE filtered). The regex `/\b[a-z0-9][a-z0-9]{2,}\b/g` requires at minimum 3 characters. ## Root causes of tonight's Jarvis-not-responding sessions

Two findings are directly responsible:

1. **CRIT-4** — mesh_runner crash loop exhausted CPU on 21+ rapid respawns between 03:21–03:23. The broken `Snell-Vern-Hybrid-Drive-Matrix` import is the trigger. Fix: add crash counter + backoff to watchdog `_start_mesh()`.

2. **HIGH-5** — Watchdog's `Get-CimInstance` query returns empty for a live Jarvis process when slow (common under load), causing `_start_jarvis()` to kill and restart Jarvis spuriously. Confirmed by log: 6 restarts tonight. Fix: require two consecutive empty polls before triggering restart.
