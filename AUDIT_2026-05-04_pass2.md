### FORGE BRIDGE — forge_listener.py

---

**AUDIT-2-001 · LOW · singleton guard substring match is fragile**

File: `C:\Xova\forge_listener.py` — singleton guard:
```python
siblings = [p for p in procs if script in (p.get("CommandLine") or "")]
if len(siblings) > 1:
    sys.exit(0)
```
Logic is correct as written: with only this process running `len(siblings) == 1`, guard does not fire. When a second instance starts `len(siblings) == 2`, guard fires for the newcomer. However, if the existing process was started with a different path format (relative vs absolute), the `script` substring match may miss it, allowing two instances to coexist.

Fix: Normalise the script path before comparison. Replace `script in (p.get("CommandLine") or "")` with `os.path.basename(script) in (p.get("CommandLine") or "")` so the basename `forge_listener.py` matches regardless of whether the path is absolute or relative.

---

**AUDIT-2-002 · HIGH · forge_listener log not rotated**

File: `C:\Xova\forge_listener.log` — opened with `open(LOG_FILE, "a")` on every write. Grows unbounded. No rotation, no max size. Will fill disk after extended operation.

Fix: Replace bare file append with Python's `logging.handlers.RotatingFileHandler`:
```python
import logging.handlers
handler = logging.handlers.RotatingFileHandler(
    LOG_FILE, maxBytes=5_000_000, backupCount=3, encoding="utf-8"
)
```
Keeps last 3 × 5 MB = 15 MB of logs, then rotates.

**AUDIT-2-003 · HIGH · queue not persisted across restarts**

File: `C:\Xova\forge_listener.py` — `_queue: list[dict]` global, populated in `_route_inbox()` enqueue branch.

If forge_listener is killed with pending items in queue they are silently lost. No read-on-startup to recover.

Fix:
- Define `QUEUE_FILE = r"C:\Xova\memory\forge_queue.json"`.
- On startup: if `QUEUE_FILE` exists, read it into `_queue`, then delete or clear the file.
- On every enqueue: write `_queue` list to `QUEUE_FILE` atomically (write to `.tmp`, rename).
- On dequeue after successful dispatch: rewrite `QUEUE_FILE` with updated list.

**AUDIT-2-004 · HIGH · forge_outbox is single-entry last-write-wins**

File: `C:\Xova\memory\forge_outbox.json` — `_deliver_reply()` writes one JSON object, overwriting any previous entry.

If Xova is not polling (window minimised, Vite HMR in progress), a reply can be overwritten by a later message before it is consumed. No message loss protection.

Fix (minimal): Add a `consumed` field and have App.tsx mark it consumed on read:
```json
{"correlation_id": "...", "text": "...", "ts": 123, "consumed": false}
```
App.tsx reads outbox, if `consumed === false` processes the message and writes back `{"consumed": true}`. forge_listener checks `consumed` before overwriting — only overwrites if `consumed === true`.

Fix (robust): Append-only `forge_outbox.jsonl` keyed by `correlation_id`; consumers mark their entry consumed by appending a `{"consumed": true, "correlation_id": "..."}` record.

**AUDIT-2-005 · HIGH · rate limit counter resets on restart**

File: `C:\Xova\forge_listener.py` — `_call_timestamps: list[float]` global, `_check_rate_limit()`.

`_call_timestamps` is in-memory only. A restart resets the counter to zero. An operator could cycle forge_listener to bypass the 20/hour cap.

Fix: Persist call timestamps to `C:\Xova\memory\forge_rate_log.json`.
- On startup: read file if exists, load `timestamps` list, prune entries where `ts < time.time() - 3600`, populate `_call_timestamps`.
- On every new call: append current timestamp, write file.
- File format: `{"timestamps": [1234567890.123, ...]}`.
- Write is atomic (write `.tmp`, rename) to avoid corrupt reads.

**AUDIT-2-006 · MEDIUM · _route_voice_to_forge advances ts unconditionally**

File: `C:\Xova\forge_listener.py` — `_route_voice_to_forge()`.

`_last_voice_ts` is updated whether or not the voice_inbox entry was addressed to forge. This is intentional (prevents reprocessing non-forge messages) but means clock drift between processes could cause a forge-bound message to be skipped if its `ts` is less than or equal to `_last_voice_ts` set by a prior non-forge entry.

Fix: Track two separate timestamps — `_last_voice_ts` for "last entry seen" (advance unconditionally) and `_last_forge_voice_ts` for "last forge-bound entry processed" (advance only on match). Route skipping should use `_last_voice_ts` for the seen check; deduplication of forge processing should use `_last_forge_voice_ts`.

**AUDIT-2-007 · MEDIUM · no retry on claude subprocess failure**

File: `C:\Xova\forge_listener.py` — `_call_claude()` error branch.

If `claude --print` exits non-zero (rate limit hit, auth error, network drop), forge_listener logs the error and the message is discarded permanently.

Fix: On non-zero exit, push the original message dict back onto `_queue` with a `_retry_count` field incremented. After 3 retries, log failure and discard. Use `threading.Timer(delay, _process_queue_item, args=[msg])` for the delay (30s / 60s / 120s) so the main poll loop is not blocked.

**AUDIT-2-008 · MEDIUM · isJarvisToForge variable name misleading**

File: `C:\Xova\app\src\App.tsx` — voice_inbox processing block.

`isJarvisToForge` is true when `(parsedRole === "jarvis" || parsedRole === "xova") && recipient === "forge"`. The name implies only Jarvis→Forge but it also covers Xova-originated messages bound for forge. Misleading for future readers.

Fix: Rename to `isRoutedToForge`. One-line rename, no logic change.

**AUDIT-2-009 · LOW · forge_listener PID not written to agent_board**

File: `C:\Xova\forge_listener.py` — `_update_board()`.

`agent_board.json` forge section contains `alive`, `last_seen`, `forge_mode`, `calls_this_hour` but not `pid`. Watchdog and diagnostics tools cannot identify which PID is the live forge_listener from the board alone.

Fix: Add `"pid": os.getpid()` to the forge board entry inside `_update_board()`.

**AUDIT-2-010 · LOW · correlation_id race in forge_outbox polling**

File: `C:\jarvis\src\jarvis\tools\builtin\ask_forge.py` — outbox poll loop.

`ask_forge.py` polls `forge_outbox.json` for a matching `correlation_id`. If two simultaneous forge requests are in flight (Jarvis sends while Xova is also waiting), the first consumer to read the outbox gets the reply regardless of correlation_id match; the second times out.

Fix (short-term): Add a lockfile `forge_outbox.lock` — consumer acquires lock before reading, releases after marking consumed. Only one reader at a time.

Fix (proper): See AUDIT-2-004 — move to an append-only outbox keyed by `correlation_id` so each consumer reads only its own reply.

**AUDIT-2-012 · MEDIUM · ask_forge timeout may exceed Jarvis LLM call timeout**

File: `C:\jarvis\src\jarvis\tools\builtin\ask_forge.py` — `WAIT_TIMEOUT_SEC = 150`.

If Jarvis's own LLM inference call timeout (set in Jarvis config or passed to Ollama) is shorter than 150s, the tool call surfaces as a Jarvis-level error before the 150s wait expires. The user sees a Jarvis timeout rather than a graceful "Forge hasn't replied yet."

Fix: Check `C:\jarvis\src\jarvis\config.py` (or equivalent) for `timeout` / `request_timeout`. If that value is < 150s, set `WAIT_TIMEOUT_SEC = that_value - 10`. Recommended safe default: `WAIT_TIMEOUT_SEC = 90`.

### XOVA UI — ChatFeed.tsx

---

**AUDIT-2-017 · LOW · forge bubble id may collide on same-millisecond replies**

File: `C:\Xova\app\src\App.tsx` — forge reply bubble construction:
```typescript
id: `forge-reply-${parsed.ts}`
```
`parsed.ts` is millisecond timestamp from `voice_inbox.json`. Two replies within the same millisecond share the same React key.

Fix:
```typescript
id: `forge-reply-${parsed.ts}-${Math.random().toString(36).slice(2, 7)}`
```

**AUDIT-2-018 · LOW · reply button prefill cuts mid-word**

File: `C:\Xova\app\src\components\ChatFeed.tsx` — reply button handler:
```typescript
const snippet = raw.length > 60 ? raw.slice(0, 60).trimEnd() + "…" : raw;
```
`slice(0, 60)` cuts at a hard character boundary with no word awareness. Snippet may end mid-word.

Fix:
```typescript
const snippet = raw.length > 60
  ? (() => {
      const cut = raw.slice(0, 61);
      const lastSpace = cut.lastIndexOf(" ");
      return (lastSpace > 20 ? raw.slice(0, lastSpace) : raw.slice(0, 60)).trimEnd() + "…";
    })()
  : raw;
```

**AUDIT-2-020 · LOW · watchdog kills forge_listener without checking process age**

File: `C:\Xova\xova_watchdog.py` — forge_listener exit branch calls `_kill(forge_pids)` for all matching PIDs.
